# Work Ticket Forms Plan

## Goal

Build a simpler ticket workflow using **one Google Form** and **one Google Sheet**.

The form collects requests from all teams. The sheet stores raw responses, creates normalized ticket rows, lets final users update only ticket status, and provides read-only views by team and status.

---

## Recommended Files

| Type | Name | Purpose |
| --- | --- | --- |
| Google Form | `Work Ticket Request Form` | Single intake form for all request types |
| Google Sheet | `Work Ticket Tracker` | Main ticket database, status workspace, and reporting views |
| Apps Script Project | `Work Ticket Tracker Script` | Sheet-bound script that creates tickets, protects columns, and refreshes views |

This setup replaces the older three-spreadsheet design:

- No separate Food spreadsheet
- No separate Transport spreadsheet
- No separate Final spreadsheet
- No manual import/export between files

## Google Form

Create one form named:

```text
Work Ticket Request Form
```

### Required Questions

| Question | Type | Required | Notes |
| --- | --- | --- | --- |
| `Request Type` | Dropdown | Yes | User selects the team/category |
| `Ticket Description` | Paragraph | Yes | Main work request |

### Dropdown Choices

```text
Food
Transport
```

More teams can be added later, for example:

```text
IT
Facilities
HR

```

### Optional Questions

Add these only if they are useful for the real workflow:

| Question | Type | Notes |
| --- | --- | --- |
| `Requester Name` | Short answer | Useful if Google account collection is disabled |
| `Requester Email` | Short answer or automatic collection | Useful for follow-up |
| `Priority` | Dropdown | Example: `Low`, `Normal`, `High`, `Urgent` |
| `Due Date` | Date | Useful for scheduling |
| `Attachment` | File upload | Useful for receipts, screenshots, documents |

## Google Sheet

Create one spreadsheet named:

```text
Work Ticket Tracker
```

Link the form responses to this spreadsheet.

### Tabs

| Tab | Editable By Users | Purpose |
| --- | --- | --- |
| `Form Responses 1` | No | Raw Google Form output |
| `Tickets` | Status column only | Main source of truth |
| `Food` | No | Read-only view of Food tickets |
| `Transport` | No | Read-only view of Transport tickets |
| `IN-PROGRESS` | No | Read-only view of active tickets |
| `DONE` | No | Read-only view of completed tickets |

The most important rule:

> `Tickets` is the only working ticket table. Generated view tabs should not be edited directly.

---

## Tickets Table

The `Tickets` tab should use this base column layout:

| Column | Description |
| --- | --- |
| `Ticket ID` | Generated by Apps Script |
| `Label` | Comes from form `Request Type` |
| `Ticket Description` | Comes from the form |
| `Date Time` | Comes from form timestamp or script timestamp |
| `Status` | Editable by final users |

Example rows:

| Ticket ID | Label | Ticket Description | Date Time | Status |
| --- | --- | --- | --- | --- |
| `FOOD-1001` | `Food` | `Order pantry restock` | `2026-06-22 09:00` | `CREATED` |
| `TRN-2001` | `Transport` | `Schedule airport pickup` | `2026-06-22 09:05` | `CREATED` |

### Status Values

```text
CREATED
IN-PROGRESS
DONE
```

### Row Colors

| Status | Row Color |
| --- | --- |
| `CREATED` | Default |
| `IN-PROGRESS` | Light orange |
| `DONE` | Light green |

## Ticket ID Rules

Apps Script should generate IDs based on `Request Type`.

| Request Type | Prefix | First Generated ID |
| --- | --- | --- |
| `Food` | `FOOD` | `FOOD-1001` |
| `Transport` | `TRN` | `TRN-2001` |

Suggested config:

```javascript
const TEAM_CONFIG = {
  Food: {
    prefix: 'FOOD',
    startNumber: 1000,
  },
  Transport: {
    prefix: 'TRN',
    startNumber: 2000,
  },
};
```

Use `LockService` when generating IDs so two users submitting the form at the same time do not receive the same ticket ID.

## Workflow

```text
User submits Work Ticket Request Form
        |
        v
Google Forms writes raw response to Form Responses 1
        |
        v
Apps Script onFormSubmit trigger runs
        |
        v
Script appends normalized row to Tickets
        |
        v
Final users update Status in Tickets
        |
        v
Script recolors rows and refreshes read-only views
```

---

## Permissions Model

### Form Users

Can submit requests only.

They do not need edit access to the spreadsheet.

### Final Users

Can open `Work Ticket Tracker`.

They should only edit:

```text
Tickets -> Status
```

### Protected Areas

Protect these columns in `Tickets`:

```text
Ticket ID
Label
Ticket Description
Date Time
```

Protect all generated view tabs:

```text
Food
Transport
IN-PROGRESS
DONE
```

---

## Generated Views

The generated tabs should be rebuilt from `Tickets`.

### `Food`

Shows rows where:

```text
Label = Food
```

### `Transport`

Shows rows where:

```text
Label = Transport
```

### `IN-PROGRESS`

Shows rows where:

```text
Status = IN-PROGRESS
```

### `DONE`

Shows rows where:

```text
Status = DONE
```

These views should be read-only. Users should always update status in the main `Tickets` tab.

## Recommended Final Design

Use this structure:

```text
Work Ticket Request Form
Work Ticket Tracker
├── Form Responses 1
├── Tickets
├── Food
├── Transport
├── IN-PROGRESS
└── DONE
```

Keep `Tickets` as the only editable ticket table.

This gives you:

- One form for all teams
- One spreadsheet to manage
- Automatic ticket IDs
- Protected generated fields
- A simple status workflow
- Read-only team and status views
